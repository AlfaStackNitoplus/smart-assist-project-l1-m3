Resources (L1 M3)

1) Mock Data

  • Maintain all test data in one place: MockData (Users, Menus, Role↔Menu, Tickets).
  • When you add a new menu item, update both menus and roleMenuMapping so it appears for the correct roles.

2) Login & Navigation Flow

  • Login currently validates using email + password from mock users and then routes based on role.
  • While navigating to dashboards, always pass id (userId) in queryParams so the dashboard can load user name & role.
  • Roles in mock data are: USER, SUPPORT_ENGINEER, ADMIN (keep these consistent across data and code).

3) Role-Based Menu Rendering

  • The side menu is generated from Role↔Menu mapping; the UI should iterate menus using @for (or *ngFor if needed).
  • Use @if / @switch when you need role-specific UI blocks (welcome text, actions, etc.).
  • Keep menu navigation routes aligned with app routing (avoid broken links).

4) Tickets: Filtering + MatTable

  • End User: filter tickets where createdByUserId matches logged in user.
  • Support Engineer: show tickets assigned to him + unassigned tickets.
  • Admin: show all tickets.
  • Keep a single displayedColumns definition so the MatTable remains consistent across roles.

5) UI Highlighting & Reusable Directive

  • Use [ngClass] for priority classes (low/medium/high) and [ngStyle] for small dynamic styles (weight, opacity, etc.).
  • Apply the hover directive (appInteractiveRow) to table rows so hover/select behavior is reusable across pages.

6) Pipes

  • Use built-in pipes where possible: date for createdAt/createdOn, uppercase/lowercase for status, number for age.
  • The custom ticketStatus pipe converts enum-like strings (e.g., IN_PROGRESS) into a readable label.
  • For debugging only, you can display objects using the json pipe (remove it for final UI if needed).

7) Quick Checks Before Final Submission

  • Verify login redirects correctly for all roles and menus update based on role.
  • Verify ticket visibility rules match the role and no empty table breaks the UI.
  • Verify hover directive works on all ticket tables.

Project Folder Structure

The application follows a feature-based and scalable folder structure, separating core components, layouts, and feature modules. 

src/
 └── app/
     ├── core/
     │   ├── components/
     │   │   ├── header/
     │   │   └── footer/
     │   ├── models/
     |  
     │
     ├── features/
     │   ├── auth/
     │   │   └── login/
     │   ├── admin/
     │   ├── support-engineer/
     │   ├── user/
     │   └── user-manual/
     │
     ├── layouts/
     │   ├── auth-layout/
     │   └── main-layout/
     │
     ├── app.routes.ts
     ├── app.component.ts
     └── app.config.ts
  • Core: Shared components like Header & Footer, and services like Auth service.
  • Features: Business modules such as Login, Admin, Support Engineer, User and User Manual
  • Layouts: Separate layouts for authentication and main app
  • Routing: Centralized route configuration in app.routes.ts
